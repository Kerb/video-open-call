# Техническое задание: Видео/Аудиозвонок в браузере (P2P WebRTC)

## 1. Общее описание

Веб-приложение для двусторонней видео- и аудиосвязи между двумя пользователями через браузер с возможностью обмена текстовыми сообщениями. Пользователи создают комнату и подключаются к ней по 6-символьному коду.

---

## 2. Требования к окружению

- **Браузеры**: Chrome 80+, Firefox 80+, Safari 14+, Edge 80+, Opera 67+, Samsung Internet 13+
- **Адаптивность**: Mobile first, все разрешения от 320px до 1920px
- **Протокол**: HTTPS обязателен (кроме localhost) — требуется для `getUserMedia`
- **Сервер**: Node.js 18+ или 20+

---

## 3. Архитектура

```
┌─────────────────────┐       WebSocket (Socket.IO)       ┌─────────────────────┐
│   Client A          │ ◄─────────────────────────────►   │   Signaling Server  │
│   (браузер)         │                                   │   (Node.js)         │
│                     │                                   │                     │
│   PeerConnection ◄──┼─────── WebRTC (SRTP/SCTP) ────────┼──► PeerConnection   │
│   Client B          │                                   │                     │
└─────────────────────┘                                   └─────────────────────┘
                                                                    │
                                                          ICE / STUN / TURN
                                                                    │
                                                      ┌─────────────┐
                                                      │  STUN/TURN  │
                                                      │   сервер    │
                                                      └─────────────┘
```

### Компоненты

| Компонент | Роль | Технология |
|-----------|------|-----------|
| **Signaling Server** | Обмен SDP/ICE, создание комнат, генерация кодов | Node.js + Socket.IO |
| **WebRTC Client** | Захват медиа, P2P соединение, рендеринг | `RTCPeerConnection`, `getUserMedia` |
| **STUN/ICE** | Определение публичного IP и порта | `stun:stun.l.google.com:19302` (бесплатный) |
| **TURN** | Релей для对称 NAT (опционально) | `twilio` или self-hosted `coturn` |

## 5. Пользовательские сценарии

### 5.1. Создание комнаты

1. Пользователь A нажимает «Создать комнату»
2. Клиент отправляет `POST /api/rooms` (или socket-событие `create-room`)
3. Сервер генерирует 6-символьный код: `[A-Z0-9]{6}` (без неоднозначных символов: `0/O`, `1/I/L`)
4. Сервер возвращает код, клиент переходит на страницу `/room/:code`
5. Пользователь A видит свой видеопоток (preview) и код комнаты, который нужно передать собеседнику

### 5.2. Подключение к комнате

1. Пользователь B нажимает «Подключиться к комнате»
2. Появляется поле ввода для 6-символьного кода
3. Пользователь B вводит код (можно ввести 6 символов, авто-форматирование uppercase)
4. Клиент отправляет `join-room {code}` на сервер
5. Сервер проверяет существование комнаты:
   - **Ошибка**: комнаты нет → показать сообщение «Комната не найдена»
   - **Успех**: пользователь B подключен → переход на `/room/:code`
6. Если в комнате уже есть 2 участника → ошибка «Комната переполнена»

### 5.3. Установка соединения (WebRTC Handshake)

После подключения обоих участников:

1. **Server → A**: `user-joined` — второй пользователь зашёл
2. **Client A**: создаёт `RTCPeerConnection`, добавляет локальный stream, создаёт offer (`createOffer`), устанавливает local description, отправляет offer серверу
3. **Server → B**: `offer {sdp}`
4. **Client B**: устанавливает remote description, создаёт answer, устанавливает local description, отправляет answer
5. **Server → A**: `answer {sdp}`
6. **A/B**: обмениваются ICE candidates через сервер (события `ice-candidate`)
7. Соединение установлено — оба видят/слышат друг друга

### 5.4. Текстовый чат

1. Во время звонка любой участник может отправить текстовое сообщение
2. Сообщение отправляется через Signaling Server (Socket.IO событие `send-message`)
3. Сервер ретранслирует сообщение второму участнику (событие `chat-message`)
4. Сообщение отображается в панели чата на экране комнаты
5. Сообщения не хранятся на сервере (только в памяти клиента во время сессии)
6. При отключении участника история чата очищается

### 5.5. Завершение звонка

- Любой участник нажимает «Повесить трубку» → отправляет `leave-room`
- Сервер уведомляет второго участника (`peer-disconnected`)
- `RTCPeerConnection` закрывается, медиа-треки останавливаются
- Пользователь возвращается на главный экран
- Если комната пуста — сервер удаляет её через N минут

## 7. Технические детали реализации

### 7.1. Signaling Server (Node.js + Socket.IO)

**События:**

| Событие (клиент → сервер) | Действие |
|--------------------------|----------|
| `create-room` | Создать комнату, вернуть код |
| `join-room {code}` | Подключиться к комнате по коду |
| `offer {sdp}` | Передать SDP offer второму участнику |
| `answer {sdp}` | Передать SDP answer первому участнику |
| `ice-candidate {candidate}` | Передать ICE candidate |
| `send-message { text }` | Отправить текстовое сообщение в комнату |
| `leave-room` | Покинуть комнату |

| Событие (сервер → клиент) | Действие |
|--------------------------|----------|
| `room-created {code}` | Комната создана, получен код |
| `user-joined {userId}` | Второй участник подключился |
| `offer {sdp}` | Получен offer |
| `answer {sdp}` | Получен answer |
| `ice-candidate {candidate}` | Получен ICE candidate |
| `chat-message { text, sender }` | Получено текстовое сообщение |
| `peer-disconnected` | Второй участник отключился |
| `room-not-found` | Комната с таким кодом не существует |
| `room-full` | В комнате уже 2 участника |

**Хранение комнат:** In-memory Map (только оперативная память, никаких внешних БД)

```typescript
interface Room {
  code: string;
  users: Map<string, Socket>;
  createdAt: number;
}
```

**Примечание:** текстовые сообщения не хранятся на сервере. Сервер только ретранслирует их второму участнику. История сообщений хранится в памяти клиента и очищается при завершении звонка.

### 7.2. WebRTC (клиент)

**Конфигурация RTCPeerConnection:**

```javascript
const config = {
  iceServers: [
    { urls: 'stun:stun.l.google.com:19302' },
    { urls: 'stun:stun1.l.google.com:19302' },
    // TURN (опционально):
    // { urls: 'turn:your-turn-server.com:3478', username: '...', credential: '...' }
  ]
};
```

**Медиа-ограничения:**

```javascript
const constraints = {
  video: { width: { ideal: 1280 }, height: { ideal: 720 }, frameRate: { ideal: 30 } },
  audio: true
};
```

**Кодекс:** Предпочтение H.264 для видео (широкая поддержка в браузерах), Opus для аудио

### 7.3. Генерация кода комнаты

- Алфавит: `ABCDEFGHJKLMNPQRSTUVWXYZ23456789` (32 символа, исключены `0/O/1/I/L`)
- 6 символов → 32^6 ≈ 1 млрд комбинаций
- Проверка уникальности на сервере (повторная генерация при коллизии)
- Код чувствителен к регистру — все uppercase
- Срок жизни кода: 30 минут без подключения второго участника

### 7.4. Автоматическое отключение

- Если второй участник не подключился за 30 минут → комната удаляется
- Если один из участников отключился от сокета → второй получает уведомление
- Если оба отключились → комната удаляется через 5 минут (grace period)
